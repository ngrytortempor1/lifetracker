# LifeTracker - 戦略的ビジョンとロードマップ

このドキュメントは、LifeTrackerの長期的なビジョンと戦略的方向性をまとめたものです。

---

## 🎯 中核となるテーゼ

LifeTrackerは、データコントロールと自動化された洞察という、現代のパーソナルデータ市場における根本的な対立を解決することにより、決定的な「パーソナルデータ・オペレーティングシステム（OS）」となるべく位置づけられています。

### 4つの原則

#### 1. The Obsidian Principle（データ主権）
すべてのユーザーデータは、ローカルのオープンフォーマットファイル（.jsonl）としてユーザーのデバイス上に存在します。クラウドは同期のためのユーティリティであり、使用の前提条件ではありません。

#### 2. The FGO Principle（究極の摩擦削減）
技術スタックとUXデザインのすべてを規定します。Electronを棄却し、KMP + ネイティブUIを選択したことは、このコミットメントの最も明確な証拠です。

#### 3. The Core Mission（相関の探求）
単純なデータ記録を超越し、すべての機能の最終目的は、ユーザーが自身のデータ内に存在する有意義な関係性を発見する手助けをすることです。

#### 4. The Flywheel Engine（コミュニティ主導の拡張性）
小規模なチームがあらゆる分析や可視化機能を構築することは不可能です。堅牢な「カーネル」を構築し、強力なプラグインAPIを通じてコミュニティが「アプリケーション」を構築する権限を与えます。

## 🏗️ 技術アーキテクチャ戦略

### 現在（Phase 1）
- **プラットフォーム**: Android（Kotlin）
- **UI**: Jetpack Compose
- **データ**: JSONL（ローカルファースト）
- **アーキテクチャ**: MVVM

### 将来（Phase 2-3）
- **マルチプラットフォーム**: KMP（Kotlin Multiplatform）
  - コアロジック共有
  - Android: Jetpack Compose
  - iOS: SwiftUI
  - Desktop: Compose for Desktop
- **同期**: E2EE（オプション）
- **プラグイン**: サンドボックス化されたAPI

### 技術判断の理由

#### なぜKMP？
- ロジック共有でメンテナンスコスト削減
- ネイティブUIで最高のパフォーマンス
- Electronより軽量（メモリ使用量を大幅削減）

#### なぜJSONL？
- 人間可読（データ主権）
- 追記専用（パフォーマンス）
- オープンフォーマット（ベンダーロックイン回避）
- 後方互換性（スキーマバージョニング）

## 🎓 競合分析と差別化

### 市場の二極化

#### 第一極（利便性/クラウド）
- Exist.io、Gyroscope、Daylio
- **強み**: 自動データ集約、シンプルなダッシュボード
- **弱み**: データロックイン、クラウド依存、サブスクリプション専用

#### 第二極（コントロール/ローカル）
- Obsidian
- **強み**: データ主権、プライバシー、拡張性
- **弱み**: 複雑なワークフロー、洞察を得るための手作業

### LifeTrackerの差別化

**両方の強みを統合**
- Obsidianのデータ主権 + Exist.ioの自動洞察
- ローカルファースト + 低摩擦
- プラグインエコシステム + 統合プラットフォーム

**新カテゴリーの創出**
「パーソナルデータOS」として、単なるアプリではなく基盤プラットフォームを目指す。

## 📝 まとめ

LifeTrackerは単なる習慣トラッキングアプリではなく、**パーソナルデータのOSになる**という野心的なビジョンを持っています。

- **短期**: 実用的なMVPで自分の生活を改善
- **中期**: プラグインエコシステムで価値を拡大
- **長期**: データ主権と自動洞察を両立するプラットフォームへ

このビジョンを実現するために、段階的かつ着実に開発を進めていきます。
